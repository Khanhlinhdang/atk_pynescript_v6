Welcome to ATK PyneScript V6

ATK PyneScript is a Python-native, declaration-first scripting runtime inspired by Pine Script. It is designed so users can create indicators, strategies, dashboards, and ATK-native graphics without writing low-level Qt or pyqtgraph code.

Best reading order: start with the contract, pick an example, then drop into the API reference only for the namespaces you need.

At a glance

What Matters Most Before You Start Coding

Indicators

Compute in build_indicator_frame. Keep build_visuals render-only. Use visual_* columns or static frame.attrs depending on whether the visual is row-based or object-based.

Strategies

Compute and signal in build_signal_frame. Map only in build_trade_frame. Prefer canonical trade-frame columns such as entry_side, entry_price, quantity, sl, and tp; build_mapped_trade_frame(...) remains the compatibility normalizer.

Graphics

Simple plots use static declarations. Object-style visuals use ctx.line, ctx.label, ctx.box, ctx.table, or ctx.atk.* bridge helpers.

Examples

The fastest route is usually to start from a shipped example, not from zero. Use the copybook for ready-to-run scripts and the manifest for coverage discovery.

Current-runtime additions

Seed Data and Collection Helpers Are Real Runtime Features

The current runtime now has practical, example-backed coverage for request.seed(...), map.*, and matrix.*. These are still partial parity surfaces compared with Pine, but they are no longer speculative or hidden implementation details.

request.seed

Use it when the script needs deterministic side-channel data such as inline seeded series, structured local payloads, or reproducible auxiliary values that should remain in the frame-builder stage.

map and matrix

Use map.* for keyed runtime bundles and matrix.* for compact 2D calculations. The docs site now treats both as supported teaching surfaces, not just footnotes in the capability matrix.

from source import indicator, input, map, matrix, request


indicator("Runtime Collections Demo", overlay=False)
window = input.int(4, title="Window", key="window", minval=2)


def build_indicator_frame(df, params=None):
    frame = df.copy().reset_index(drop=True)
    size = max(int((params or {}).get("window", window) or window), 2)

    seeded_source = {
        "seeded_close": frame["close"].rolling(size, min_periods=1).mean().round(4).tolist(),
    }
    frame["seeded_close"] = request.seed(seeded_source, "seeded_close")

    settings = map.new()
    map.put(settings, "window", size)
    map.put(settings, "bias", "bullish")
    frame.attrs["map_keys"] = map.keys(settings)

    grid = matrix.new(size, 2, 0.0)
    return frame

Architecture

Runtime Model

ATK PyneScript is not a Pine parser. Scripts must be valid Python. Pine-like APIs are exposed as Python functions and namespaces.

What ATK does

  • Records declarations, inputs, visuals, alerts, and library imports into module context.
  • Compiles declaration modules into managed wrappers based on BaseIndicator, BaseSubIndicator, or BaseStrategy.
  • Builds a managed visual snapshot and routes it through ATK visual adapters.
  • Lets users author visual logic without touching Qt scene code directly.

What ATK does not do

  • It does not execute Pine syntax that is invalid Python.
  • It does not promise full TradingView parity for all display or intrabar semantics.
  • It does not want heavy compute inside build_visuals() or build_trade_frame().
  • It does not require users to implement low-level graphic items for common visual workflows.

Declarations

indicator, strategy, library

indicator(title: str, **kwargs) -> dict

Registers indicator metadata into module context and controls whether the compiled wrapper becomes a main-chart overlay or a sub-panel indicator.

Important kwargs

  • overlay
  • shorttitle
  • precision
  • timeframe
  • description
  • default_styles
  • behind_chart
  • explicit_plot_zorder
  • scale
  • max_lines_count, max_boxes_count, max_labels_count, max_polylines_count, max_bars_back

ATK behavior: overlay=True compiles to BaseIndicator. overlay=False compiles to BaseSubIndicator.

strategy(title: str, **kwargs) -> dict

Registers strategy metadata and preserves execution-related flags for diagnostics and runtime behavior.

Important kwargs

  • overlay
  • shorttitle
  • precision
  • calc_on_every_tick
  • calc_on_order_fills
  • process_orders_on_close
  • fill_orders_on_standard_ohlc
  • max_bars_back
  • timeframe
  • description
  • default_styles

Current note: process_orders_on_close has the strongest practical runtime behavior today. Tick-level semantics remain partial.

library(title: str, **kwargs) -> dict

Registers a publishable declaration library whose exported functions are discovered from module callables defined in the same file.

Useful kwargs

  • version
  • overlay
  • description

Libraries can later be loaded with library_import(...) or import_library(...).

How to choose indicator

Choose indicator(...) when the script's primary output is visual analysis: lines, bands, markers, zones, tables, or dashboards. If the script does not need to emit executable trade intents, keep it as an indicator even if it contains buy or sell hints.

How to choose strategy

Choose strategy(...) when the script should produce mapped entry and risk fields that the ATK execution pipeline can consume. A strategy may still draw visuals, but strategy identity is defined by build_signal_frame and trade mapping.

How to choose library

Choose library(...) when you want reusable calculation helpers, validation helpers, or shared signal logic that multiple scripts import. A library should package functions, not chart-facing builder stages.

Parameter Where it matters How to choose it
titleUser-facing script identityPick the readable name users should see in menus and script lists. Keep it stable once the script is published.
overlayIndicator and strategy panel placementUse True when the visual belongs on the price chart. Use False for oscillators, histograms, and sub-panel diagnostics with a separate scale.
shorttitleTight UI surfacesUse it when the full title is long or repetitive. Keep it recognizable because legends and compact widgets may show only the short title.

default_stylesInitial visual appearanceUse it when a multi-plot script needs a predictable starting style. Avoid burying core logic in style defaults; keep style changes cosmetic.

Reuse

Library Imports

library_import(name, version=None, exports=None, alias=None)

Resolves a published ATK PyneScript library, validates exports, records metadata into script context, and returns a namespace object containing the selected callables.

import_library(spec, exports=None, alias=None)

Convenience wrapper over library_import. Use it when the library spec already encodes the name and version information you want.

Behavior notes: library imports are version-aware, exported callables are discovered from the target module, missing exports raise errors, and a library cannot import itself.
Current scope: published libraries can export callables and enum types. Selective imports such as Pyne Enum Utils@1::TrendMode,resolve_length are supported in plain Python files through import_library(...). Quoted import syntax is also supported, but only after the editor preprocesses the source into valid Python. 
from source import library, import_library

Most important contract

Builder Contracts and Stage Ownership

Indicator contract

  • Required: indicator(...).
  • Required: build_indicator_frame(df, params) or equivalent frame builder path used by the runtime.
  • Optional: static plot declarations such as plot, hline, fill, plotshape, plotcandle.
  • Optional: build_visuals(frame, params=None, ctx=None) for object-style or custom ATK bridge visuals.

Strategy contract

  • Required: strategy(...).
  • Required: build_signal_frame(df, params).
  • Recommended: build_trade_frame(signal_df, params, styles) returning build_mapped_trade_frame(signal_df).
  • Optional: visual declarations or build_visuals(...), but only for rendering already-prepared data.
Stage What belongs here What must not happen here
build_indicator_frame All indicator series computation, source resolution, warmup handling, row-oriented visual_* prep, static attrs config. Final drawing calls through ctx.*.
build_signal_frame Signal logic, filter logic, canonical trade-frame execution fields, strategy visuals prep. Order schema normalization beyond mapping-only helpers.
build_trade_frame Mapping-only normalization, usually build_mapped_trade_frame(...). ta.*, request.*, rolling windows, dataframe grouping, signal recomputation.
build_visuals Render-only mapping into ctx.* or ctx.atk.*. Mutating the frame, computing TA, storing stale full-frame dynamic payloads in frame.attrs.

How data should flow

Inputs -> params merge -> frame builder -> visual mapping. Inputs define defaults. Runtime params override them. The frame builder creates plain dataframe columns. Only after those columns exist should declarations or build_visuals consume them.

Why stage separation matters

When compute, trade mapping, and rendering are mixed together, the script becomes hard to debug, hard to slice, and hard for the runtime to keep consistent across chart reloads. The stage split is not style advice; it is the stability contract.

Library contract

  • Required: library(...).
  • Recommended: export small, reusable helpers or enum types.
  • Not expected: build_indicator_frame, build_signal_frame, or chart visuals.
  • Import through import_library(...) or library_import(...).

What a library should optimize for

Keep library code deterministic, narrow, and reusable. It should own helper logic such as length resolution, bias labeling, transform helpers, or shared signal utilities, not chart rendering or strategy mapping.

If you need to... Put it here Reason

Compute EMA, ATR, BBands, MACD, or MTF trend filtersbuild_indicator_frame or build_signal_frameAll TA and request work belongs in compute stages.

Important: If you use frame.attrs for object-like visuals, store only static config there and derive dynamic x, y, tail, or slice-dependent values from the actual frame passed into build_visuals().

Runtime logic

Indicator, Strategy, and Library Processing Flows

Indicator flow

Use this when the script computes analysis series and then renders them through static declarations or a render-only build_visuals(...).

Strategy flow

Use this when the script emits canonical trade-frame fields that ATK will normalize into execution intents for backtest, replay, and live runtimes.

Library flow

Use this when the script exists only to export reusable functions, constants, or enum types for other PyneScript modules.

indicator(...)
  -> input.* defaults register into module context
  -> build_indicator_frame(df, params)
  -> frame columns + optional static attrs config
  -> static plot declarations and or build_visuals(frame, params, ctx)
  -> managed visual snapshot
  -> ATK visual router and adapters
strategy(...)
  -> input.* defaults register into module context
  -> build_signal_frame(df, params)
  -> entry_* / sl / tp / trail_* prepared
  -> build_trade_frame(signal_df, params, styles)
  -> build_mapped_trade_frame(signal_df) or equivalent mapping-only normalization
  -> normalized trade frame
  -> execution intent model
  -> backtest / replay / live execution path
library(...)
  -> exported callables and enum types discovered from the module
  -> publish metadata and dependency reports generated
  -> importing script calls import_library(...) or library_import(...)
  -> selected exports resolve into a namespace object
  -> indicator or strategy script reuses the helpers in its own builders

How defval works

defval is the value returned immediately at import time and the value used when the runtime has not supplied an override. Treat it as the script's safe startup configuration.

Why key matters

key is the stable identity for the setting in the runtime. Keep it short, explicit, and durable. Changing keys later breaks continuity between saved settings and the script contract.

How to merge params

Use inputs as typed defaults, then merge params over them inside the frame builder. That keeps the script runnable as plain Python while still respecting runtime overrides.

Technical analysis

ta.* Namespace

Most-used first Start with sma, ema, rsi, atr, crossover, crossunder.
Multi-output helpers Use bbands, macd, stoch, kdj when the script needs grouped outputs.
Keep it in builders All ta.* work belongs in build_indicator_frame or build_signal_frame, never in build_visuals or trade mapping.

Core helpers

sma(series, length), ema(series, length), rsi(series, length=14), atr(frame, length=14), williams(frame, length=14), true_range(frame), psar(frame, acceleration=0.02, maximum=0.2).

Multi-output helpers

bbands(series, length=5, std=2.0, mamode="sma") returns upper, basis, lower. macd(...) returns MACD, signal, histogram. stoch(...) returns K and D. kdj(...) returns K, D, J.

Signal helpers

crossover(a, b) and crossunder(a, b) return boolean series and already handle shifted-bar comparisons.

    source_series = pd.to_numeric(frame[str(merged.get("source_type", "close"))], errors="coerce") if str(merged.get("source_type", "close")) in frame.columns else pd.to_numeric(frame["close"], errors="coerce")
    upper, basis, lower = ta.bbands(
        source_series,
        length=max(int(merged.get("length", 20) or 20), 2),
        std=float(merged.get("multiplier", 2.0) or 2.0),
        mamode=str(merged.get("mamode", "sma") or "sma").lower(),
    )
    frame["lb"] = lower.reset_index(drop=True)
    frame["cb"] = basis.reset_index(drop=True)
    frame["ub"] = upper.reset_index(drop=True)
    return frame

Multi-timeframe

request.* Namespace

Use only when needed Reach for request.* only when the script truly depends on higher-timeframe or alternate-frame data. It adds state and alignment complexity quickly.
Keep it in signal or indicator builders Do all request work in build_indicator_frame or build_signal_frame, never in build_trade_frame or build_visuals.
Start from the MTF example If this is your first multi-timeframe script, copy the confirmation strategy pattern before trying to improvise resampling logic.

request.security(frame, timeframe, expr, fill_method="ffill")

Resamples the source frame to the requested timeframe, evaluates expr on the resampled frame, and aligns the result back to the base frame.

  • frame: source dataframe
  • timeframe: string such as 15m, 1h, 1d
  • expr: callable or series expression
  • fill_method: ffill or bfill

def build_signal_frame(df: pd.DataFrame, params: dict | None = None) -> pd.DataFrame:
    frame = df.copy().reset_index(drop=True)
    p = {
        "fast_period": int(fast_period),
        "slow_period": int(slow_period),
        "trend_length": int(trend_length),
        "confirm_tf": str(confirm_tf),
    } | dict(params or {})
    ema_fast = ta.ema(frame["close"], int(p["fast_period"]))
    ema_slow = ta.ema(frame["close"], int(p["slow_period"]))
    htf_trend = request.security(
        frame,
        str(p["confirm_tf"]),
        lambda x: ta.ema(x["close"], int(p["trend_length"])),
    )
    frame["buy_signal"] = ta.crossover(ema_fast, ema_slow).fillna(False) & (frame["close"] >= htf_trend.fillna(frame["close"]))
    frame["sell_signal"] = ta.crossunder(ema_fast, ema_slow).fillna(False) & (frame["close"] <= htf_trend.fillna(frame["close"]))
    frame["entry_side"] = np.where(frame["buy_signal"], "BUY", np.where(frame["sell_signal"], "SELL", ""))
    frame["entry_price"] = frame["open"]
    frame["quantity"] = float(trade_qty)
    frame["size_pct"] = 0.0
    return frame

How to think about plot(...)

plot does not compute anything. It declares, "when the frame contains a column with this name, render it with this identity and style." The first argument should therefore be the exact output column name, not the formula itself.

Render stage

build_visuals(frame, params, ctx)

Render only This stage should only translate prepared data into visual intents. If you are still calculating values here, the script shape is wrong.
Choose between ctx and ctx.atk Use ctx.line, ctx.label, ctx.box for object namespaces. Use ctx.atk.* for ATK-native composite renderers.
Beware stale attrs payloads Static config can live in frame.attrs, but slice-dependent coordinates and dynamic values should be derived from the frame passed into this function.
Rule: build_visuals(...) must stay render-only. It should map already prepared data into visual intents. It should not compute indicators, mutate the frame, or build stale slice-dependent attrs payloads.

ctx.line

Create and mutate line objects with new, set_xy1, set_xy2, set_color, etc.

ctx.label, ctx.box, ctx.polyline, ctx.linefill, ctx.table

Object namespaces work through recorded visual intents, so you can create dashboards, overlays, and object bundles without touching low-level renderer code.

ctx.gfx.item

Generic graphics intent entry point, usually wrapped by more specific ATK bridge helpers.

Question Recommended answer
Should this visual be a dataframe column or an object payload?If one bar naturally corresponds to one visual value, use columns. If the visual is a box, table, multi-point path, or grouped bundle, use object APIs in build_visuals.
Should I pass raw x/y numbers or chart.point objects?Use raw row-aligned columns for static visuals. Use chart.point for object APIs when coordinates should anchor by index or time with explicit object semantics.
Should I read from frame.attrs?Yes for static object config such as colors, headers, labels, and shape bundles. No for stale, slice-dependent dynamic coordinates that should be recalculated from the provided frame.
Should this function return something?Return object handles only if the creation path benefits from it. Many scripts simply call ctx.* methods for side-effect-like intent recording and return None.

What ctx really is

ctx is the runtime's visual intent gateway. You do not draw pixels directly here. You describe objects, updates, and bridge payloads, and the ATK visual layer turns those intents into actual rendered items.

What frame should contain by this stage

By the time build_visuals runs, the dataframe should already contain every computed series and every row-aligned visual column needed by the renderer. If you still need TA, rolling windows, or MTF calls here, the script design is off by one stage.

How to Build Indicators

Overlay line indicator

Use indicator(..., overlay=True), compute series in build_indicator_frame, and declare one or more plot calls.

Sub-panel oscillator

Use overlay=False, compute oscillator series, declare hlines for thresholds, and optionally add ATK histograms or custom panels in build_visuals.

Object-rich visual indicator

Prepare static payload in frame.attrs, then render with ctx.box, ctx.label, ctx.line, or ctx.atk.*.

Recommended indicator design workflow

  1. Choose overlay or sub-panel first.
  2. Define all inputs explicitly with stable key names.
  3. Resolve source columns safely if the source can vary.
  4. Compute series in build_indicator_frame.
  5. If visuals are row-oriented, add visual_* columns.
  6. If visuals are object-like, store static config in frame.attrs.
  7. Render only in build_visuals.

size_pctRelative position size.Use it when the script expresses sizing as a percent of capital, portfolio, or policy budget rather than absolute units.
sl and tpRisk exits.Compute them in the signal frame from ATR, structure, volatility, or fixed multiple logic. They are part of strategy intent, not a trade-frame afterthought.
trail_offsetTrailing stop distance.Use it only when the strategy is genuinely trailing. Do not emit it as a placeholder value just because the field exists.
tag, comment, and alert_messageHuman-readable metadata.Use them to preserve context such as signal family, filter state, or reason code that will help debugging later.

Minimal good strategy pattern

Compute signals, compute risk, emit only the canonical trade-frame columns you actually need, and return build_mapped_trade_frame(signal_df). The cleanest strategies are usually the easiest ones to trust.

Common beginner mistake

Many first drafts compute signals in build_signal_frame and then recompute ATR, stop levels, or even direction inside build_trade_frame. That duplicates logic and breaks the stage contract. Keep strategy thinking in one place.

Best Practices and Anti-Patterns

Do this

  • Compute everything in the appropriate frame builder.
  • Use visual_* columns for row-aligned visuals.
  • Use frame.attrs only for static object config.
  • Use stable key values for visuals and inputs.
  • Prefer build_mapped_trade_frame for strategies.
  • Use import_library(...) inside raw .py scripts and reserve quoted imports for editor-preprocessed source.
  • Quote examples from source/example_user_scripts so users can paste and run them immediately.

Do not do this

  • Do not compute TA inside build_visuals.
  • Do not compute strategy logic inside build_trade_frame.
  • Do not store dynamic, slice-sensitive payloads entirely inside frame.attrs.
  • Do not teach users outdated examples that hide stage boundaries.
  • Do not fix runtime parity problems only inside user examples when the correct fix belongs upstream.
